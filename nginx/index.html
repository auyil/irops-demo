Challenge & Opportunities

Irregular operations are inevitable;
their impact is not.

Irregular Operations (IROPS) — storms, mechanical faults, ATC restrictions — trigger cascading passenger disruptions that require rapid, complex decision-making. This prototype demonstrates how autonomous AI agents can resolve these events in seconds, not hours, within strict regulatory and safety guardrails.

WHY THIS USE CASE
The Trigger

The Ripple Effect

A single cancelled flight doesn't just disrupt one passenger — it cascades through connecting flights, crew legality windows, gate allocations, and catering chains. Hub-and-spoke networks amplify every failure.

The Constraint

Operational Complexity

Every rebooking decision must respect IATA delay codes, Australian Consumer Law refund rights, loyalty tier priority rules, and strict SSR safety protocols — simultaneously, at scale, in real time.

The Opportunity

Multi-Agent Orchestration

The problem decomposes naturally into specialist agents: one fetches data, one executes rebooking logic, one handles customer communication. Each has a bounded silo and explicit guardrails.

The Architecture Signal

Skill Libraries + Governance

Policy lookup, IATA code classification, and SSR safety checking are reusable skills — not hardcoded logic. Model selection, token budgets, and temperature controls are explicit governance artefacts.

System Design

Built for the real constraints
of enterprise AI.

This prototype is not only a proof of concept — it is a reference architecture. Every design decision reflects how AI should be deployed in a regulated, safety-critical, customer-facing environment: with explicit governance, observable behaviour, and the kind of guardrail layering that lets product teams move fast without breaking trust.

Design Philosophy
The Guardrail

Standards & Governance

Every AI decision is constrained, logged, and explainable. Model selection, token budgets, temperature settings, and safety rules are defined in a single governance configuration — not scattered across code. The system cannot spend more than it's allowed to, and it cannot take actions it hasn't been authorised for.

The Pattern

Builder Infrastructure

The architecture is modular by design. Agents own clearly bounded silos. Skills are reusable across agents. The orchestrator coordinates without doing data work. This is a reference pattern product teams can adopt, not a one-off script — swap the database connection string and it runs on Redshift in production.

The Domain

Domain Expertise as Code

Aviation regulatory constraints — IATA delay codes, ACL consumer rights, SSR safety protocols, tier priority rules — are not buried in prompts. They are explicit technical artefacts: guardrail hooks, skill libraries, and semantic layers that encode domain knowledge in a form that can be audited, tested, and updated independently of the model.

Agent Architecture

Four agents. Two workflows. One governance model.

The system separates concerns at the agent level. The Orchestrator routes and coordinates — it never touches data. The Fetcher owns all reads. The Rebooking agent owns all writes. The Concierge handles all customer interaction. Each agent has its own model, system prompt, and tool set, and can be tested, replaced, or upgraded independently.

This is the subagent pattern in practice: silo the work, not just the prompts.

Skill Library

Skills for documents. Subagents for decisions.

The distinction matters. A skill is a stateless, deterministic function — no model call, no latency, no cost. It encodes reference knowledge (policy documents, IATA code tables, SSR definitions) in a form that any agent can call without re-deriving the same information from scratch.

Subagents own judgment and action. Skills own knowledge and lookup. Keeping these separate means policy updates propagate instantly across all agents without prompt engineering — change the policy chunk, all agents benefit.

SKILL lookup_policy
Keyword search over 30 chunks parsed from Qantas Compensation Policy, Conditions of Carriage, and ACCC Travel Rights. Returns the exact policy section relevant to a compensation decision.
Used by: Rebooking · Concierge
SKILL lookup_iata_code
Maps an IATA delay code to its human label and control classification (controllable / uncontrollable). This single lookup determines voucher eligibility for every passenger in every disruption.
Used by: Rebooking · GuardrailHook
SKILL check_ssr_codes
Parses a passenger's Special Service Request codes and returns safety sensitivity flags. A single WCHR or UMNR code immediately halts automated rebooking and routes the case to a human agent.
Used by: Concierge · GuardrailHook
Guardrail Architecture

Two layers. Different jobs. Both necessary.

A single guardrail layer is insufficient for a safety-critical system. Infrastructure-level guardrails handle generic threats. Application-level guardrails handle domain-specific compliance. They operate at different points in the request lifecycle and are maintained by different teams.

Layer 1 — Infrastructure

Bedrock Guardrails

Provisioned via AWS CDK · Applied before agent invocation

Classifies raw user input against seven denied topic categories using natural language understanding. Blocks prompt injection attempts, off-topic queries (new bookings, loyalty points, competitor airlines), and harmful content — before the agent is ever invoked. Configured by the platform team, not the product team.

prompt_injection new_booking seat_upgrade baggage_claims loyalty_points general_travel competitor_airlines
Layer 2 — Application

GuardrailHook

Strands BeforeToolCallEvent · Fires before every tool call

Domain-specific constraints that require knowledge of passenger state, IATA codes, and business policy — things a generic content classifier cannot evaluate. Three distinct rules, each addressing a different category of risk.

TC-02 · SSR SAFETY HALT
Detects safety-sensitive Special Service Request codes (WCHR, UMNR, MEDA, OXYG) before rebook_flight executes. Halts automation, flags for human agent. Aircraft compatibility must be verified by a person.
TC-03 · POLICY ALIGNMENT
Checks IATA delay code control classification before issue_voucher. Blocks hotel vouchers for uncontrollable events (weather, ATC). Enforces Qantas compensation policy section 3 at the tool level.
TC-04 · BUDGET CAP
Checks cumulative workflow cost against the configured USD cap before every tool call. Terminates the agent loop if exceeded, preserving partial resolutions. Prevents runaway spend in high-volume disruptions.
Observability & Cost Monitoring

Every token. Every tool call. Every decision.

The WorkflowTraceHook captures every agent event — tool calls, invocation completions, guardrail interventions — and streams them to the frontend via Server-Sent Events in real time. This is not logging after the fact; it is live observability built into the agent lifecycle.

The TokenLedger accumulates cost per agent, per model, per workflow. The cost bar in the demo updates after every tool call. This is the same pattern that would feed a production CloudWatch dashboard or cost allocation report.

Data architecture: demo-ready, production-portable.

The demo runs on SQLite in a container — zero infrastructure, instant startup. The schema mirrors a production Airline Passenger Service System (PSS) exactly: the same five tables, the same IATA column conventions, the same foreign key relationships.

The FetcherAgent's semantic layer — DDL, business glossary, and six gold-standard JOIN queries — is injected as a structured system prompt. This is the Amazon Q Generative SQL pattern: teach the model the schema once, let it query accurately without hallucinating column names. Swap the connection string for Redshift Serverless and the pattern is production-ready.

Infrastructure & Delivery
▲

AWS CDK (Python)

All infrastructure is code. EC2 instance, IAM roles, Bedrock Guardrail, EventBridge scheduler, CloudFront distribution, and S3 bucket — all provisioned from a single CDK project. Deploy order and dependency management are explicit.

◈

Bedrock + Strands SDK

Amazon Bedrock provides model invocation (Claude Haiku 4.5, Nova Lite) and managed Guardrails. The AWS Strands SDK provides the agent loop, hook system, and tool execution framework — no custom orchestration code required.

◉

Docker + nginx

Two containers managed by docker-compose: nginx (static files + reverse proxy) and FastAPI (agent API). EC2 instance profile provides AWS credentials — no secrets in configuration. Containers restart automatically on EC2 start via systemd.

◎

EventBridge Scheduler

EC2 starts at 7am and stops at 7pm Sydney time, weekdays only — timezone-aware, DST-handled automatically. Manual override via AWS CLI when needed. Reduces compute cost by ~70% versus always-on.
